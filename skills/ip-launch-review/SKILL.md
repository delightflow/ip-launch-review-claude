# IP Launch Review

## Overview

공개 전 IP 전략 검토 스킬. 해커톤 발표, 데모데이, 투자자 미팅, 외주 핸드오프, 공개 런치 직전에 **현재 사용자의 프로젝트**를 분석해 특허 후보, 브랜드 리스크, 발명신고서 초안, 7일 액션 플랜을 제공한다.

**핵심 원칙:** 이 스킬 레포가 아니라 사용자의 현재 작업 디렉토리(프로젝트)를 분석 대상으로 삼는다. 추측이 아닌 실제 코드·파일·로직 분석 기반으로 작성한다.

## When to Use

- 해커톤 발표 직전
- 데모데이·투자자 미팅 전
- 기술 스펙·코드를 외주에 넘기기 전
- 공개 런치·앱스토어 제출 전
- 핵심 워크플로우가 담긴 덱(deck) 공유 전

---

## How to Run

현재 작업 디렉토리의 핵심 파일(알고리즘, 파이프라인, 데이터 구조, API 스키마 등)을 직접 읽고 아래 순서로 분석한다.

---

### 1. 프로젝트 스냅샷

- 서비스 한 줄 요약
- 주요 사용자 타겟
- 핵심 기술 스택 (언어·프레임워크·AI 모델 등)
- 배포 형태 (웹앱/모바일/API/CLI 등)

---

### 2. 브랜드 / 상표 관점

- 서비스 이름의 상표 등록 가능성 (기술적·설명적 표현 여부, 식별력)
- 유사 상표 충돌 가능성 (동일·유사 업종 기준)
- 도메인·SNS 핸들 선점 여부 확인 권고
- 네이밍 리스크 및 개선 방향

---

### 3. 특허 후보 — 발명 포인트 상세 분석

우회가 어렵고 독립적으로 청구 가능한 기술 흐름을 3개 이상 추출한다.
**각 발명 포인트는 아래 4-Box 구조**로 작성한다:

```
[발명 N] 발명 제목
─────────────────────────────────────────
① 해결 과제 (종래 기술의 문제점)
   - 기존 방식에서 무엇이 불편하거나 비효율적인가?
   - 사용자/시스템이 겪는 구체적 한계

② 핵심 구성 (해결 수단 — 실제 코드/로직 기반)
   - 이 발명이 구체적으로 어떻게 동작하는가?
   - 핵심 데이터 흐름, 알고리즘 단계, 파라미터 구조
   - 관련 파일/함수/클래스 참조

③ 발명의 효과
   - 이 방식으로 얻는 정량적·정성적 이점
   - 기존 대비 개선 수치나 특성 (빠름/정확함/자동화 등)

④ 독립청구항 핵심 요소 (변리사 전달용 뼈대)
   - [구성요소 A]: ...
   - [구성요소 B]: ...
   - [구성요소 C]: ...
   (명세서 작성 전 변리사가 청구항 범위를 설계할 때 참고)
```

---

### 4. 선행기술 리스크 예측

각 발명 포인트별로:

| 발명 | 유사 기술 카테고리 | 충돌 가능성 | 회피 방향 |
|------|-------------------|------------|-----------|
| 발명 1 | | 높음/중간/낮음 | |
| 발명 2 | | 높음/중간/낮음 | |
| 발명 3 | | 높음/중간/낮음 | |

- **충돌 높음**: 검색엔진·특허청 공개 DB 선행기술 조사 권고 후 출원 여부 판단
- **충돌 중간**: 청구항 범위를 구체적으로 좁혀 차별화 가능
- **충돌 낮음**: 우선 가출원으로 선점 후 정식출원 준비

---

### 5. 쉽게 복제될 수 있는 부분

- UI·랜딩 페이지·일반적인 AI 프롬프트 등
- 경쟁사가 1~2주 안에 따라 만들 수 있는 요소
- 특허보다 빠른 실행(선점)이 더 유효한 부분

---

### 6. 지금 공개하지 않는 것이 좋은 부분

아래 항목이 발표자료·데모·외주 문서에 포함되어 있다면 제거 권고:

- [ ] 핵심 알고리즘 로직 / 수식 / 가중치
- [ ] 프롬프트·에이전트 라우팅·폴백 구조
- [ ] 내부 랭킹·스코어링 규칙
- [ ] 학습 데이터 출처·전처리 파이프라인
- [ ] API 키·설정값이 포함된 스키마
- [ ] 유일한 데이터 구조 또는 DB 스키마

### 8. 가출원 vs 정식출원 판단

아래 기준으로 각 발명의 출원 트랙을 제안한다:

**가출원 즉시 권고** (아래 중 2개 이상 해당):
- 공개 예정일이 2주 이내
- 핵심 알고리즘이 이미 작동하는 상태
- 경쟁사 유사 제품이 이미 존재
- 투자자 미팅·데모데이 발표 예정

**정식출원 직행 권고** (아래 중 2개 이상 해당):
- 충분한 시간적 여유 있음 (3개월+)
- 발명 구성이 이미 상세하게 정리됨
- 선행기술 조사 완료 상태

| 발명 | 권고 트랙 | 이유 |
|------|-----------|------|
| 발명 1 | 가출원 / 정식출원 | |
| 발명 2 | 가출원 / 정식출원 | |
| 발명 3 | 가출원 / 정식출원 | |

### 9. 변리사 미팅 전 준비 사항

변리사 상담 전에 아래 항목을 정리해 갈 것:

1. **메인 청구항 축**: 어떤 기술 단계를 핵심 청구항으로 삼을 것인가?
2. **분할 출원 가능성**: 하나의 출원으로 묶을지, 복수로 분리할지
3. **공개 데드라인**: 발표·런치·논문 게재 등 공개 예정일
4. **관련 선행기술 파악 여부**: 내가 아는 유사 특허·제품 목록
5. **비용 범위**: 가출원만 or 정식출원까지 예산 범위

## Pre-Disclosure Checklist

결과 말미에 아래 체크리스트를 포함한다:

- [ ] 서비스 이름이 상표로 선점 가능한가?
- [ ] 도메인·SNS 핸들을 먼저 확보했는가?
- [ ] 핵심 기능 중 우회가 어려운 구조는 정리했는가?
- [ ] 발표자료에서 비공개 권고 항목을 제거했는가?
- [ ] 지금 공개하면 바로 복제될 부분을 파악했는가?
- [ ] 발명별 발명신고서 초안을 완성했는가?
- [ ] 선행기술 충돌 가능성을 검토했는가?
- [ ] 가출원 vs 정식출원 트랙을 결정했는가?
- [ ] 변리사 미팅 전 준비 항목을 정리했는가?
